Notes de version de NetBeans IDE 6.8

Dernière mise à jour : July 10, 2013

NetBeans IDE est un environnement de développement intégré (EDI) modulaire basé sur des normes, écrit dans le langage de programmation Java. Le projet de NetBeans IDE consiste en un EDI Open Source complet écrit dans le langage de programmation Java et en une plate-forme d'application cliente riche, qui peut être utilisée comme structure générique pour créer n'importe quel type d'application. Pour savoir comment installer le logiciel sur votre système, reportez-vous à la section NetBeans IDE 6.8 Installation Instructions. Pour en savoir plus sur les nouvelles fonctionnalités incluses dans cette version de l'EDI, reportez-vous à la page NetBeans IDE 6.8 Release Information.

Nouveautés de la version 6.8

NetBeans IDE 6.8 est une mise à jour de NetBeans IDE 6.7.1 qui présente les modifications suivantes :

  • Prise en charge de Java EE 6
  • Prise en charge de JSF 2.0 avec de nombreuses nouvelles fonctionnalités
  • Prise en charge de PHP 5.3 avec le framework Symfony
  • Intégration améliorée de la fonctionnalité Kenai et prise en charge du suivi des bogues Jira
  • Améliorations supplémentaires incluses dans NetBeans IDE 6.8

Pour plus d'informations sur cette version, reportez-vous à la page Informations de version sur NetBeans IDE 6.8.

Informations sur la prise en charge multilingue

NetBeans IDE est proposé dans les langues prise en charge par Sun suivantes : portugais brésilien (BR), japonais (JP) et chinois simplifié (ZH).

Des traductions de l'EDI ont été effectuées par la communauté dans plusieurs autres langues. Vous pouvez les télécharger à partir de la section Community Contributed (contributions de la communauté) du menu déroulant Langue de l'IDE.

Compatibilité

Lorsque vous exécutez l'EDI pour la première fois, vous pouvez importer un sous-ensemble des paramètres que vous avez utilisés dans les versions précédentes de NetBeans IDE. Si vous choisissez de ne pas importer de paramètres d'une version précédente, l'EDI s'ouvre avec les paramètres par défaut.

Remarque : l'EDI reconnaît uniquement les installations précédentes si le répertoire utilisateur se trouve à l'emplacement par défaut. Il ne reconnaît pas les installations pour lesquelles vous avez spécifié un répertoire utilisateur à l'aide du commutateur --userdir. Si vous souhaitez importer les paramètres à partir d'un EDI que le programme d'installation ne reconnaît pas, lancez l'EDI et utilisez la fonction d'importation de la fenêtre Options. Après avoir cliqué sur Importer dans la fenêtre Options, utilisez le bouton Parcourir et recherchez le répertoire utilisateur (userdir) d'installation de l'EDI contenant les paramètres que vous voulez importer. Pour plus d'informations sur l'emplacement du répertoire utilisateur de NetBeans IDE, reportez-vous à la FAQ What is a userdir and where can I find it? (en anglais).

Remarque : si vous modifiez dans NetBeans IDE 6.8 les propriétés d'un projet créé dans NetBeans IDE 4.1 ou une version précédente, le projet ne fonctionnera plus dans les versions précédentes de NetBeans IDE.

Problèmes connus : programme d'installation

Les problèmes non résolus concernant le programme d'installation pour cette version sont les suivants :

  • Problème n° 177872 : sous Mac OS X, l'add-on GlassFish v3 et les outils de mise à jour risquent de ne pas fonctionner correctement

    Description : sous Mac OS X, l'add-on basé sur pkg et les outils de mise à jour ne fonctionnent pas correctement avec les comptes non root si GlassFish v3 a été installé par un utilisateur root.

    Solution : utilisez la commande chown -R pour changer le propriétaire de l'installation GlassFish et sélectionnez l'utilisateur non root. Par exemple : sudo chown -R <newuser>:admin /Applications/NetBeans/glassfish-v3-b73, où newuser est l'utilisateur non root. Une fois cette commande utilisée, l'utilisateur peut exécuter directement les outils bin/updatetool et bin/pkg.

  • Problème n° 6720021 : le programme d'installation ne fonctionne pas lorsque vous utilisez OpenSolaris avec n'importe quelle version du JDK 5.

    Description : le programme d'installation ne fonctionne pas lorsque vous utilisez OpenSolaris avec n'importe quelle version du JDK.

    Solution : effectuez l'installation avec le JDK 6 ou exécutez la commande export AWT_TOOLKIT=XToolkit avant d'exécuter le programme d'installation avec le JDK 5.

  • Problème n° 93568 : sous Solaris et Linux, l'installation se bloque lorsque la connexion réseau est interrompue.

    Description : sur les plates-formes Solaris et Linux, si la connexion réseau est interrompue, l'installation s'arrête lorsque le programme d'installation recherche un JDK compatible dans les répertoires réseau.

    Solution : restaurez la connexion réseau et continuez l'installation.

  • Problème n°117172 : une fenêtre vierge s'affiche lors de l'exécution du programme d'installation sous Ubuntu.

    Description : le programme d'installation affiche une boîte de dialogue blanche ou grise si vous l'exécutez sous Ubuntu et que Beryl/Compiz/Compiz Fusion est installé.

    Solution : exécutez la commande export AWT_TOOLKIT=MToolkit avant d'exécuter le programme d'installation.

  • Problème n°123300 : Sous Fedora Core 8, le programme d'installation n'affiche plus l'interface et retourne sur la console.

    Description : sous Fedora Core 8, l'affichage de l'interface d'installation s'arrête et le système retourne sur la console.

    Solution : reportez-vous à https://netbeans.org/servlets/ReadMsg?list=nbusers&msgNo=98765.

  • L'assistant d'installation de NetBeans IDE 6.8 ne démarre pas correctement (se bloque) lorsque l'utilisateur interrompt manuellement la procédure d'installation et que le programme d'installation indique qu'une autre instance du programme d'installation est en cours d'exécution.

    Description : sous Windows, lorsque Java6 est utilisé par défaut et que des fichiers zip volumineux sont stockés sur le bureau, l'assistant d'installation ne démarre pas correctement (se bloque) lorsque l'utilisateur interrompt manuellement la procédure d'installation et que le programme d'installation indique qu'une autre instance du programme d'installation est en cours d'exécution.

    Solution : avant l'installation, supprimez tous les fichiers zip volumineux du bureau.

Problèmes connus : EDI principal

Les problèmes non résolus pour cette version sont les suivants :

  • Difficultés de démarrage de l'EDI (Solaris x86 et Dual-Core Opteron).

    Description : sur certaines configurations de Solaris x86 et Dual-Core Opteron, vous pouvez rencontrer des difficultés pour démarrer l'EDI en raison d'un bogue JDK lié au chargement des bibliothèques look&feel GTK.

    Solution : si tel est le cas, spécifiez l'option de ligne de commande suivante : -laf javax.swing.plaf.metal.MetalLookAndFeel.

  • Les caractères ne s'affichent pas correctement dans certains environnements linguistiques asiatiques.

    Description : lorsque vous exécutez l'EDI avec certains environnements linguistiques asiatiques sous Ubuntu et certaines autres distributions Linux, le texte de l'écran de l'EDI risque de devenir illisible. Sur certaines distributions Linux, les caractères à plusieurs octets s'affichent sous la forme d'un rectangle en raison de problèmes de paramètres des polices JDK.

    Solution : consultez la FAQ pour savoir comment résoudre ce problème : http://wiki.netbeans.org/FaqI18nLinuxAsianFonts

  • Problème n° 6720021 : difficultés de démarrage de l'EDI sous OpenSolaris avec n'importe quelle version du JDK 5.

    Description : il est possible que l'EDI ne démarre pas sous OpenSolaris si vous utilisez une version du JDK 5.

    Solution : utilisez JDK 6 ou exécutez la commande export AWT_TOOLKIT=XToolkit avant de démarrer l'EDI.

  • Problème n° 64840 : communication réseau interrompue sous Linux lorsque le protocole IPv6 est activé.

    Description : la communication réseau est interrompue sous Linux lorsque le protocole IPv6 est activé, ce qui empêche l'EDI de se connecter aux serveurs intégrés, à la fonctionnalité de mise à jour automatique, etc.

    Solution : si cela se produit, exécutez l'EDI à l'aide du commutateur -J-Djava.net.preferIPv4Stack=true'. Vous pouvez également désactiver le protocole IPv6 (dans /etc/modules.conf ou /etc/modprobe.conf).

    Pour en savoir plus, reportez-vous au problème n° 5087907.

  • Problème n° 167401 : l'EDI se bloque de façon aléatoire sur les systèmes MacOS X qui utilisent le JDK 1.6.0 Update 13.

    Description : il est possible que l'EDI se bloque de façon aléatoire lorsque vous parcourez la barre de menu principale en raison du bogue du JDK d'Apple n° 6993269.

    Solution : vous pouvez utilisez une autre version du JDK ou modifier les options de démarrage par défaut du fichier netbeans.conf sur -J-Dapple.laf.useScreenMenuBar=false.

  • Problème n°110074 : l'éditeur se bloque régulièrement sous Solaris.

    Description : lorsque vous exécutez l'EDI sous Solaris avec le JDK 6, 6u1 ou 6u2, les threads peuvent se bloquer pendant un long moment et provoquer l'arrêt de l'éditeur.

    Solution : mettez l'installation de votre JDK 6 à niveau vers l'Update 3 ou une version plus récente.

  • Problème n° 88776 : il est possible que les applications profilées se bloquent si le profilage s'effectue en mode de rattachement dynamique.

    Description :lors d'une exécution sur le JDK 1.6, le profilage d'une application en mode de rattachement dynamique peut provoquer le blocage de l'application profilée.

    Solution : lancez l'application que vous souhaitez profiler à l'aide du commutateur -Xshare:off.

    Voir aussi le problème n°113847.

  • Problème n°119617 : le curseur se perd dans l'éditeur et la saisie au clavier est impossible.

    Description : il peut parfois arriver qu'après avoir utilisé une indication dans la fenêtre de l'éditeur, le point d'insertion dans l'éditeur disparaisse et que vous ne puissiez plus utiliser le clavier. Ce problème a principalement été signalé sous Mac OS X 10.4, mais également sous certaines versions d'Ubuntu.

    Solution : utilisez la souris pour ouvrir et fermer les boîtes de dialogue (une boîte de dialogue de propriétés par exemple). Le focus est ainsi replacé sur l'éditeur, et le point d'insertion et le clavier fonctionnent à nouveau normalement.

  • Problème n°130357 : I18N - Mac OS ne génère pas de javadoc si le nom du projet ou le chemin d'accès comporte plusieurs octets.

    Description : lors de la création du fichier de paramètres temporaire, la tâche javadoc est écrite dans le codage du système d'exploitation par défaut, c'est-à-dire MacRoman. Il semble s'agir d'une situation Mac OS dans laquelle les parties BSD et Mach (appels système et noyau) acceptent les paramètres en UTF-8, mais où le système d'exploitation et HFS restent configurés comme MacRoman.

    Solution : évitez d'utiliser plusieurs octets dans le nom du projet ou dans le chemin d'accès au projet.

  • Problème n°152916 : difficultés de démarrage de l'EDI sous Mac OS X 10.5.5 64 bits à l'aide du JDK 1.6_07.

    Description : sur certaines configurations Mac OS X 10.5.5 64 bits avec JDK 1.6_07 défini comme plate-forme Java par défaut du système, vous pouvez rencontrer des difficultés à démarrer l'EDI depuis le Finder ou à partir de la ligne de commande.

    Solution : définissez les paramètres par défaut du système sur JDK 1.5.

  • Problème n° 150416 : impossible de créer la base de données d'exemple sans InnoDB.

    Description : la base de données d'exemple dans NetBeans utilise des clés étrangères. Le moteur MyISAM ne prend pas en charge les clés étrangères. Par conséquent, il n'est pas possible d'utiliser des bases de données d'exemple sans le moteur de stockage InnoDB.

    Solution : installez le moteur de stockage transactionnel InnoDB.

  • Problème n°158421 : problèmes d'utilisation de l'EDI avec G1 Garbage Collector.

    Description : si vous utilisez G1 Garbage Collector (dans JDK 6 Update 14 ou une version plus récente et dans le JDK 7), des exceptions peuvent être émises lorsque vous utilisez l'EDI.

    Solution : n'utilisez pas G1 Garbage Collector (supprimez -XX:+UnlockExperimentalVMOptions -XX:+UseG1GC à partir de netbeans.conf ou de la ligne de commande).

Problèmes connus : Java ME

NetBeans vous permet de développer des applications Java ME pour les périphériques MIDP, CLDC et CDC.

Les problèmes non résolus concernant Java ME pour cette version sont les suivants :

  • Linux, Solaris et d'autres systèmes d'exploitation ne sont pas pris en charge par le SDK 3.0 de Java ME.

    Description : NetBeans 6.8 Mobility et le SDK 3.0 de Java ME fournis avec cette version ne prennent en charge que le système d'exploitation Windows.

    Solution : le programme d'installation Linux est intégré Sun Java Wireless Toolkit 2.5.2. Les utilisateurs de Mac OS peuvent télécharger le Java Platform Micro Edition Software Development Kit 3.0 for MacOS EA afin de développer des applications mobiles. La FAQ comporte des instructions concernant l'installation de Java ME SDK 3.0 for MacOS dans l'EDI.

  • Problème n° 139031 : l'émulateur ne démarre pas sous Windows si le fichier Java binaire n'est pas dans le chemin d'accès de l'environnement.

    Description : le SDK de Java ME et d'autres émulateurs ne se lancent pas sous Windows si Java.exe ne se trouve pas dans le chemin d'accès.

    Solution : installez le JDK à l'aide des paramètres par défaut ou, si vous effectuez une installation personnalisée, n'oubliez pas de sélectionner le JRE public pour l'installation. Si le JDK est déjà installé, vous pouvez ajouter le fichier Java.exe dans les paramètres du chemin d'accès de votre système.

Problèmes connus : Ruby

NetBeans vous permet de développer des applications Ruby et Ruby Rails.

Les problèmes non résolus concernant Ruby pour cette version sont les suivants :

  • Problème n° 171338 : les utilisateurs ne peuvent pas créer de projets Rails.

    Description : les utilisateurs ne peuvent pas créer de projet lorsque le chemin d'accès au répertoire de la plate-forme JRuby intégrée contient des espaces.

    Solution : mettez manuellement RubyGems à niveau à partir de la ligne de commande : nb_install_dir_ruby2/jruby-1.4/bin/jruby -S gem update --system

Problèmes connus : C/C++

NetBeans C/C++ prend en charge les applications C et C++ sous Microsoft Windows, Mac OS X, Linux et le système d'exploitation SolarisTM. Les développeurs C/C++ peuvent intégrer des applications existantes dans le système de projet NetBeans ou développer de nouvelles applications et bibliothèques. La prise en charge C/C++ de NetBeans couvre l'intégralité du cycle de développement : les projets et modèles, un éditeur compatible avec les langages avancé, des versions basées sur le fichier makefile et un débogage.

Pour plus d'informations sur l'installation et la configuration des compilateurs et des outils pour la prise en charge NetBeans C/C++, reportez-vous à la section Installing and Configuring C/C++ Support.

Les problèmes non résolus concernant C/C++ pour cette version sont les suivants :

  • Problème n° 149053 : sur les plates-formes Windows, le chemin d'accès de création d'outils ne peut pas contenir d'espace.

    Description : sur les plates-formes Windows, si le chemin permettant de créer des outils contient des espaces, il est tronqué et les outils sont introuvables.

    Solution : n'utilisez pas d'espaces dans le chemin de création des outils.

  • Problème n° 166811 : GDB échoue sur Open Solaris lorsque vous commencez à déboguer des fichiers C++.

    Description : lorsque vous déboguez des projets C++ sur Open Solaris, GDB échoue en raison d'une erreur de segmentation.

    Solution : supprimez les packages gcc-432 comme le décrit la solution à ce problème.

  • Problème n° 148204 : sur les plates-formes Windows, la saisie utilisateur ne fonctionne pas dans la fenêtre de résultat lors du débogage.

    Description : sur les plates-formes Windows, lorsque vous déboguez une application et que le type de console du projet est défini sur la fenêtre de résultat, vous ne pouvez pas saisir de données dans cette fenêtre. Ce problème n'affecte pas le développement avec un hôte distant.

    Solution : définissez le type de console sur Terminal Externe. Pour ce faire, cliquez avec le bouton droit de la souris sur la fenêtre Projets, sélectionnez le nœud Exécuter dans la boîte de dialogue Propriétés du projet, choisissez Terminal Externe dans la liste déroulante Type de console, puis cliquez sur OK.

  • Problème n° 121776 : il est parfois difficile de déboguer le code compilé Sun Studio avec gdb.

    Description : sur les systèmes d'exploitation Solaris et Linux, le code compilé avec les compilateurs Sun Studio ne peut pas toujours être correctement débogué avec le débogueur gdb. La plupart des problèmes connus sont résolus dans le logiciel Sun Studio 12 mais certains autres existent. Si vous ne parvenez pas à déboguer des programmes compilés avec les compilateurs Sun Studio 12, appliquez d'abord les derniers patchs du produit Sun Studio 12. Si le problème persiste, signalez le problème dans Issuezilla et rendez cette tâche dépendante du problème.

    Au fur et à mesure que des problèmes sont découverts, nous ouvrons de nouveaux cas et actualisons cette tâche pour qu'elle dépende de ces problèmes, ce qui donne une idée de l'état actuel du débogage des programmes compilés avec les compilateurs Sun Studio 12 à l'aide de NetBeans et du débogueur gdb.

    Ces problèmes n'ont pas d'incidence sur le débogage du code compilé avec les compilateurs Sun Studio 12 à l'aide de l'EDI Sun Studio 12 et du débogueur dbx.

  • Problème n°121775 : le Xcode le plus récent est nécessaire pour déboguer sur Tiger (Mac OS X 10.4).

    Description : lorsque vous déboguez sur Macintosh avec Tiger (Mac OS X 10.4), la version par défaut des compilateurs gcc et g++, génère un code accompagné d'un bogue dans le résultat du débogage. Apple a sorti un package Xcode actualisé (version 2.5) pour résoudre ce problème. Les versions appropriées de gcc, g++ et gdb sont : gcc: 4.0.1 - Apple build 5367 g++: 4.0.1 - Apple build 5367 gdb: 6.3.50 - Apple build 573 (10/20/2006).

    Pour télécharger cette version de Xcode, rendez-vous sur la page http://developer.apple.com/tools/download.

    Ce problème ne concerne pas Leopard (Mac OS X 10.5).

    Les symptômes du problème sont les suivants : les exécutables compilés avec l'indicateur -gdwarf (la valeur par défaut pour les projets de l'application C/C++) ont des informations de ligne incorrectes et les points d'arrêt peuvent être ignorés.
