Notas de versão do IDE NetBeans 6.5

Última atualização: August 9, 2012

O IDE NetBeans é um ambiente de desenvolvimento integrado, modular e baseado em padrões, escrito na linguagem de programação Java. O projeto do NetBeans consiste em um IDE de código aberto escrito na linguagem de programação Java e em uma plataforma de aplicativos , que pode ser utilizada como uma estrutura genérica para construir qualquer tipo de aplicativo.

    • O que há de novo no 6.5

    O NetBeans IDE 6.5 é uma atualização significativa do NetBeans IDE 6.1 e inclui as seguintes alterações:

    • Suporte a PHP.
    • Suporte melhorado a JavaScript.
    • Suporte melhorado a banco de dados: histórico SQL, auto-completar com SQL e visualização e edição aperfeiçoadas dos resultados.
    • Construtor de GUI: suporte a Nimbus e nomes de classe simples.
    • JUnit: suporte a método de teste único.
    • Depurador: recriação do recurso Passar dentro.

    Para informações sobre os recursos principais de desenvolvimento no NetBeans IDE, consulte a página de Informações sobre o NetBeans IDE 6.5.

    Compatibilidade


    Ao executar o IDE pela primeira vez, você pode importar um subconjunto de configurações usadas em uma versão anterior do NetBeans IDE. Se você optar por não importar as configurações de uma versão anterior, o IDE abrirá com as configurações padrão.

    Nota: o IDE reconhece somente as instalações anteriores onde o diretório do usuário reside na localização padrão. O IDE não reconhece as instalações nas quais você especificou um diretório de usuário usando o switch --userdir. Se desejar importar as configurações de um IDE que o instalador não reconhece, baixe uma versão de arquivamento do IDE, em vez de usar o instalador.

    Nota:se você editar as Propriedades do Projeto no IDE NetBeans IDE 6.5, de um projeto criado no IDE NetBeans 4.1 ou anterior, não será mais possível trabalhar com esse projeto em versões anteriores do IDE NetBeans.

    Disponibilidade do módulo UML

    Os editores de diagrama UML foram totalmente reescritos nessa versão. A complexidade dos novos recursos atrasou o desenvolvimento que teve como resultado as seguintes alterações:

    • O UML não é o instalador do NetBeans 6.5. O UML foi movido do instalador para a Central de Atualizações e pode ser instalado através do menu Ferramentas > Plug-ins no IDE. O módulo UML também está disponível em formato de arquivo zip.
    • Suporte a UML no 6.5. São suportados cinco tipos de diagramas: atividade, classe, seqüência, estado e caso de uso. Esses diagramas são atualizados automaticamente nos projetos UML anteriores ao 6.5 quando são salvos na nova versão do módulo UML.

    Problemas conhecidos: instalador

    Os problemas do instalador não-resolvidos desta versão são os seguintes:

    • Problema nº 93568: no Solaris e no Linux, a instalação pára quando a conexão de rede é interrompida.

      Descrição: nas plataformas Solaris e Linux, se a conexão de rede é interrompida, a instalação pára quando o instalador está buscando um JDK compatível nos diretórios de rede.

      Solução alternativa: restabeleça a conexão de rede e prossiga com a instalação.

    • Problema nº 117172: uma janela em branco ao executar o instalador no Ubuntu.

      Descrição: o instalador é exibido como uma caixa de diálogo em branco/cinza ao ser executado no Ubunty com Beryl/Compiz/Compiz Fusion
      instalado.

      Solução alternativa: execute o comando export AWT_TOOLKIT=MToolkit antes de executar o instalador.

    • Problema nº 119861, Problema nº 117334: a instalação é interrompida com um erro quando o comando unpack200 é executado.

      Descrição: a instalação é interrompida com um erro durante a execução do comando unpack200. Esse erro pode ser causado pelo aplicativo Logitech Process Monitor (Webcam Effects Helper).

      Solução alternativa: se você tem o aplicativo Logitech Process Monitor (Webcam Effects Helper) (LVPrcSrv.exe) em execução no sistema, desligue-o antes da instalação ou desinstalação

    • Problema #122030, Problema #130368: No Fedora Core 8 e no Ubuntu 8.04 alpha 6, a instalação falha quando o JDK 7 é usado.

      Descrição: no Fedora Core 8 e Ubuntu 8.04 alpha 6, a instalação de GlassFish V 2 falha quando o JDK 7 é usado (JDK7 IcedTea 64-Bit Server VM).

      Solução alternativa: para estas plataformas, use o JDK 5 ou o JDK 6.

    • Problema nº 123300. No Fedora Core 8, o instalador interrompe a exibição da interface e volta ao console.

      Descrição: no Fedora Core 8, a exibição da interface de instalação é interrompida e o sistema volta ao console.

      Solução alternativa: consulte https://netbeans.org/servlets/ReadMsg?list=nbusers&msgNo=98765.

    • A página de Download do NetBeans IDE 6.5 pode ser exibida no idioma selecionado no menu de Escolha de Idioma da Página.

      Descrição: A página do Download é exibida no idioma especificado como padrão do navegador. Se você estiver usando um navegador não localizado, será exibido em Inglês. Para visualizar a página de Download em Chinês, Japonês, ou Português Brasileiro, selecione o idioma desejado na lista de idiomas da Página de Seleção de Idioma, no topo da página de download.

      Solução alternativa: As versões localizadas em Chinês, Japonês e Português Brasileiro no navegador Firefox permitem a abertura da página de Download localizada. Para visualizar a página de Download em Chinês, Japonês, ou Português Brasileiro, selecione o idioma desejado na lista de idiomas da Página de Seleção de Idioma, no topo da página de download.

    • O assistente de instalação do NetBeans IDE 6.5 não inicia corretamente (trava). Depois que o usuário interrompe o processo de instalação manualmente, o programa de instalação informa que outra instância de instalação está em execução.

      Descrição: No Windows, quando o Java6 é utilizado como padrão e grandes arquivos compactados são armazenados na área de trabalho, o assistente de instalação não inicia devidamente (trava). Depois que o usuário interrompe o processo de instalação manualmente, o programa de instalação informa que outra instância de instalação está em execução.

      Solução Alternativa: Antes da instalação, remova todos os grandes arquivos compactados da área de trabalho.

    Problemas conhecidos: IDE do núcleo

    Os problemas não-solucionados desta versão são os seguintes:

    • Dificuldades para iniciar o IDE (Solaris x86 e Dual-Core Opteron).

      Descrição: em certas configurações do Solaris x86 e do Dual Core Opteron, você poderá encontrar problemas para iniciar o IDE devido a um bug do JDK ao carregar as bibliotecas GTK look&feel.

      Solução alternativa: se você enfrentar esse comportamento, especifique a seguinte opção de linha de comando:-laf javax.swing.plaf.metal.MetalLookAndFeel

    • Os caracteres não são exibidos adequadamente em alguns locais asiáticos.

      Descrição: ao se executar o IDE em alguns locais asiáticos no Ubuntu e em algumas outras distribuições de Linux, o texto da tela do IDE pode se tornar ilegível. Em determinadas distribuições do Linux, os caracteres multibytes podem ser exibidos em forma de retângulo devido a problemas de configuração de fonte do JDK.

      Solução alternativa: como não há nenhuma solução alternativa para esse problema, recomenda-se uma distribuição de Linux alternativa, caso você utilize um local sujeito a esse comportamento.

    • Problema nº 64840: comunicação de rede interrompida no Linux com IPv6 ativado.

      Descrição: comunicação de rede interrompida no Linux com IPv6 ativado, impedindo a conexão do IDE com servidores empacotados, atualização automática etc.

      Solução alternativa: se isso acontecer, execute o IDE usando a chave -J-Djava.net.preferIPv4Stack=true'. Como alternativa, desative o IPv6 (em /etc/modules.conf ou /etc/modprobe.conf).

      Para obter mais informações, consulte:Problema nº 5087907.

    • Problema nº 62919: aplicativo com perfil pode falhar ao se tirar um instantâneo de memória.

      Descrição:na criação de perfil para aplicativos, o aplicativo com perfil poderá falhar quando você tentar tirar um instantâneo de memória.

      Solução alternativa: inicie o aplicativo para o qual você quer traçar um perfil com a chave -Xnoclassgc.

    • Problema nº 110074: o editor congela periodicamente no Solaris.

      Descrição: ao executar o IDE no Solaris com JDK 6, 6u1 ou 6u2, os segmentos podem ficar bloqueados por um longo período de tempo e fazer com que o editor não responda.

      Solução alternativa: migre sua instalação de JDK 6 para a Atualização 3 ou mais recente.

    • Problema nº 88776: aplicativo com perfil traçado pode falhar quando a criação de perfil é realizada no modo de conexão dinâmica.

      Descrição:ao executar no JDK 1.6, a criação de perfil em um aplicativo com modo de conexão dinâmica pode levar o aplicativo a falhar.

      Solução alternativa: inicie o aplicativo para o qual você deseja criar um perfil com o switch -Xshare:off.

      Consulte também: Problema nº 113847.

    • Problema #6389282 do JDK 6 : Aplicativos Java criados com o modelo de projeto Java Desktop Application, podem não funcionar nos idiomas Chinês e Japonês quando o JDK 6 for usado.

      Descrição: Aplicativos Java criadas com o modelo de projeto Java Desktop Application com, podem não funcionar nos locais onde o Linux, nos idiomas Japonês e Chinês, quando o JDK 6 estiver instalado.

      Solução alternativa: use o JDK 5.0 ou execute os projetos de locais em inglês.

    • Problema #119617: O cursor desaparece no editor e não é possível utilizar o teclado.

      Descrição: As vezes, depois de usar uma dica na janela do editor, o ponto de inserção no editor desaparece e o teclado não funciona. Isto foi observado com maior freqüência no Mac OS X 10.4, mas houveram alguns relatos em algumas versões do Ubuntu.

      Solução alternativa:Use o mouse para abrir e fechar a caixa de diálogo (por exemplo, uma janela de propriedades). Isto faz com que o foco volte para o editor e o ponto de inserção e o teclado funcionará normalmente.

    • Problema #119567: Projeto para aplicativo de banco de dados da área de trabalho não funciona se o nome ou no caminho do projeto houver um caractere não-ASCII .

      Descrição: A causa está nas bibliotecas atuais de toplink utilizadas no IDE. Um problema com a biblioteca toplink foi registrado. Este problema com o Glassfish foi corrigido no Glassfish v2.1. Entretanto, o NetBeans 6.5 suporta o glassfish v2 u2, e a correção não existe nesta versão do Glassfish.

      Solução alternativa: Use somente caracteres ASCII no nome, ou no caminho, de um projeto de um aplicativo de banco de dados da área de trabalho.

    • Problema #118174 e outros problemas relacionados: I18N - Javadoc não encontrado no Solaris, talvez em outras plataformas, em certos casos ao utilizar o navegador Firefox.

      Descrição: Ao usar o Firefox no Solaris, e as vezes no Mac OS e no Windows, se caracteres multibyte ou non-ascii forem usados no nome ou no caminho do projeto, e se as propriedades de codificação não estiverem ajustadas para utf-8 (o que significa que estes caminhos estão naquela codificação), o javadoc criado não pode ser encontrado ou exibido.

      Por exemplo: No caso de um projeto que tem euc-jp como propriedade de codificação do projeto no Solaris, e o nome ou o caminho do projeto usa caracteres Japoneses daquela codificação, o javadoc não será encontrado e uma mensagem de erro será exibida no Firefox.

      O problema é que parte da URL é UTF-8 e parte é EUC-JP; a ferramenta do javadoc armazena a URL na codificação gerada para a página da web, mas o caminho correto é gerar o texto na codificação exigida (neste caso, a EUC-JP) e as URLs em UTF-8.

      Solução alternativa: Evite o uso de multibyte no nome do projeto ou no caminho do projeto.

    • Problema #130357: I18N - Mac OS não gera o javadoc se o nome ou caminho do projeto conter multibyte.

      Descrição: A tarefa javadoc, durante a criação de um arquivo de parâmetros temporários, é escrita na codificação padrão do OS que é a MacRoman. Esta parece ser uma situação do Mac OS, onde as partes BSD e Mach (system & kernel calls) aceitam parâmetros em UTF-8, mas o OS e HFS se divulgam como MacRoman.

      Solução alternativa: Evite o uso de multibyte no nome do projeto ou no caminho do projeto.

    • Problema #130311: No Mac OS, algumas mensagens ou cadeias de caracteres em outro idioma ou conjunto de caracteres não são exibidos na janela de saída corretamente.

      Descrição: Por exemplo, no Mac OS X 10.4. ao usar System.out.println em Java, as cadeias que contém caracteres japoneses são exibidas corretamente no código fonte, mas são exibidas como pontos de interrogação na janela de saída.

      Solução alternativa: Não existe solução alternativa até o momento.

    • Problema nº 144388: adicionar argumento para instalação de servidor MySQL independente

      Descrição: se o usuário utilizar o servidor MySQL independente com o IDE, não é exibida nenhuma saída na janela Saída do IDE.

      Solução alternativa: adicionar o argumento --console à lista de argumentos para iniciar o servidor.

    • Problema nº 144398: SQL CC: CC não aparece nas tabelas do sistema MySQL

      Descrição: após executar select * from information_schema.statistics; e, em seguida, select * from seguido de Ctrl+Barra de espaço, não aparece nenhuma tabela do sistema na lista de CC.

      Solução alternativa: use o driver Connector/J no MySQL 5.1.7.

    • Problema nº 150416: não é possível criar o DB de exemplo sem o InnoDB

      Descrição: o banco de dados de exemplo do NetBeans usa chaves externas. O mecanismo MyISAM não oferece suporte a chaves externas. Portanto, os bancos de dados de exemplo não podem ser usados sem o mecanismo de armazenamento InnoDB.

      Solução alternativa: instale o mecanismo de armazenamento InnoDB transacional.

    Problemas conhecidos: Web e Java EE

    Os problemas não-resolvidos da Web e Java EE para esta versão são os seguintes:

    • Kit Visual Web JSF Backwards Compatibility
      Descrição: o kit Visual Web JSF Backwards Compatibility fornece bibliotecas suplementares exigidas pelo projetista Visual Web para oferecer suporte a projetos baseados em J2SE 1.4 ou J2EE 1.4. Essas bibliotecas não têm código-fonte aberto e, portanto, são distribuídas como um plug-in separado do NetBeans. O kit instalará as seguintes bibliotecas exigidas para projetos J2EE 1.4:
      • JavaServer Faces 1.1 Reference Implementation (RI)
      • Rowset Reference Implementation (RI)
      • Bibliotecas JAX-RPC do JWSDP 1.6

      Para obter o kit de Compatibilidade Retroativa, no menu principal, escolha Ferramentas > Plug-ins.

    • Notas de versão dos componentes JSF do Woodstock 4.1 e 4.2
      Descrição: as notas de versão dos componentes JSF do Woodstock 4.1 e 4.2 se encontram no site Project Woodstock, no endereço http://java.net/projects/woodstock. A versão que acompanha o IDE NetBeans 6.5 é a Woodstock 4.2 Build 3.
    • Não existe suporte para Ajax Sample Components
      Descrição: Esta versão não suporta os componentes Blue Prints Ajax Sample, porque elas são baseadas em uma versão antiga do Dojo, que não é mais compatível com as versões mais recentes. Componentes Woodstock incluídos no NetBeans requerem versões mais recentes dos APIs Dojo para renderização no cliente. Para a maioria dos componentes Ajax BP popularmente utilizados (suportados no Netbeans 5.5), existe uma funcionalidade equivalente disponível no Projeto woodstock, conforme listado abaixo.
      • Preenchimento automático - Este blog (http://blogs.sun.com/dmitry/entry/creating_autocomplete_entry_field_with ) descreve como alcançar uma funcionalidade parecida com os componentes da biblioteca do Projeto Woodstock.
      • Calendário 'Pop up' - o Projeto Woodstock inclui um componente de Calendário.
      • Selecione o Valor de campo - Use auto completar em seu lugar, que possui uma funcionalidade parecida mas uma UI diferente.
      • Barra de progresso - Projeto Woodstock inclui um componente de barra de progresso.
      • Mapas Google - Planejado para o futuro.
      • Componente de pontuação - Planejado para o futuro.
      • Editor de Rich Text - Não suportado
    • Problema nº 113888: os temas do Woodstock são incompatíveis com a migração de projetos do NetBeans IDE 5.5/5.5.1 para o NetBeans 6.5. Esse problema está relacionado com o Problema nº 99063, elemento de corpo ausente alternando para novo tema do Woodstock.

      Descrição: os projetos com temas personalizados e migrados para o IDE NetBeans 6.5 produzirão erros de elemento de corpo ausente.

      Solução alternativa: alterne manualmente para o tema padrão antes de migrar. Temas personalizados não funcionarão e precisarão ser reconstruídos.

    • Problema 94193: não é possível editar propriedades de bean na janela Tópicos ou por meio de pacotes de código-fonte na janela Projetos.

      Descrição: no IDE NetBeans 6.5, o item Adicionar menu fica desativado na janela Navigator e não há nenhuma funcionalidade para editar as propriedades por meio de pacotes de código-fonte.

      Solução:

      1. Clique duas vezes em SessionBean1 (ou em outro arquivo java) para abrir o bean no editor do Java.
      2. Adicione a propriedade manualmente, como private String CustomerId;.
      3. Clique com o botão direito do mouse na propriedade e escolha Gerar código.
      4. No menu pop-up Gerar, selecione Getters e setters.

      5. Na caixa de diálogo Gerar getters e setters de SessionBean1, selecione a propriedade e pressione OK.

        Os métodos getter e setter públicos são inseridos após a propriedade.

    • Problema nº 95022: O redimensionamento do campo de texto e de outros componentes não funciona.

      Descrição: Devido a uma mudança nos componentes do Woodstock, comandos de redimensionamento foram removidos dos seguintes componentes Woodstock: TextField, Text Area,Password Field, Drop Down e List Box. Esse problema afeta especialmente os projetos importados do Java Studio Creator e do NetBeans Visual Web Pack 5.5/5.5.1. Os componentes redimensionados podem ter aparência diferente de versões anteriores.

      Solução alternativa: para os componentes TextField e TextArea, use os atributos de colunas e linhas, respectivamente. A caixa suspensa e de listagem foram melhoradas para incluir uma nova propriedade de largura.

    • Quando exibidos no IE7, todos os componentes de botão podem ser processados com uma largura incorreta.

      Descrição: quando um botão é colocado com o posicionamento absoluto, sua largura fica incorreta. Por exemplo, quando o IDE NetBeans 6.5 usar o posicionamento absoluto para colocar componentes no Modo de Layout de Grade, esse problema poderá ocorrer.

      O IE7 espera um valor a ser especificado para a largura dos botões. Se um atributo de largura não for especificado, a largura será definida como largura:automática, o que resultará na definição da largura do botão em toda a largura disponível do elemento de recipiente do botão. Por exemplo, se o estilo do botão for esquerda=130px, então o botão começará em 130px à esquerda e se expandirá até a largura do elemento <body> .

      Solução alternativa: existem várias maneiras diferentes de resolver esse problema no NetBeans:

      • Especifique a propriedade de largura do botão ou redimensione o botão no IDE para fazer com que o IDE atribua uma largura para o botão.
      • No IDE, solte o botão no componente Grupo de painéis para colocar o botão dentro de uma marca de span. Neste caso, não é preciso definir o posicionamento quando o IDE define o posicionamento do componente Grupo de painéis. Por exemplo, o código JSP de um botão dentro do componente Grupo de painéis pode ser o seguinte: 
        <webuijsf:panelGroup binding="#{Page1.groupPanel1}" id="groupPanel1" style="position: absolute; left: 240px; 
top: 168px">
            <webuijsf:button binding="#{Page1.button1}" id="button1" text="Button"/>
</webuijsf:panelGroup>
    • Problema nº 106299: faces-config.xml de projetos Visual Web JSF não contém mais a seção de local, como em versões anteriores.

      Descrição: faces-config.xml tinha a seção comentada relacionando o modo como indicar outros locais, com algumas linhas para certos locais como exemplo. Os usuários agora devem adicionar essas linhas manualmente, verificando se o nome das localidades de seu interesse estão na seção de localidades aceitas.

      Solução alternativa: adicione manualmente a localidade desejada à seção de localidades do arquivo faces.config.xml. Aqui está um exemplo de como o código deve aparecer:

      <application>
             <locale-config>
                 <default-locale>en</default-locale>
                 <supported-locale>localidade desejada</supported-locale>
             </locale-config>
         </application>

      Consulte também Problema nº 103085

    • Problema #125965: Exceção de ponteiro nulo em Arrastar e soltar depois de fechar um projeto de Visual Web.

      Descrição: Ao trabalhar com dois projetos Visual Web JSF, e ao fechar o segundo projeto e retornar ao primeiro, isso pode causar uma exceção de ponteiro nulo para a primeira ação de arrastar e soltar.

      Solução alternativa: Pressione atualizar no designer e continue trabalhando. Se isso não ajudar, reinicie o IDE.

Problemas conhecidos: Java ME


O NetBeans 6.5 permite que você desenvolva aplicativos baseados em Java ME para dispositivos MIDP, CLDC e CDC.

Os problemas do Java ME não solucionados nesta versão são os seguintes:

  • O desenvolvimento de aplicativos CDC (Connected Device Configuration) exige a instalação de um simulador de plataforma CDC.

    Descrição: O Sun Java Toolkit 1.0 para CDC não faz parte do pacote da versão 6.5.

    Solução alternativa: baixe o Sun Java Toolkit 1.0 for CDC a partir da Sun Developer Network. O NetBeans CDC Emulator Platform Setup Guide tem instruções para a instalação deste e de outros SDKs para desenvolvimento de CDC.

  • O módulo Java ME não será executado se um nome de diretório, um caminho para o diretório de instalação ou um diretório de usuário contiver caracteres não-ASCII.

    Descrição: o Sun Java Wireless Toolkit não oferece suporte a caracteres não-ASCII em caminhos de projeto.

    Solução alternativa: certifique-se de que o diretório de usuário não contenha caracteres não-ASCII. Se contiver, mude o diretório de usuário padrão do IDE.

  • Problema nº 6389282 do JDK 6: locais de Linux em chinês e japonês com o JDK 6 não executam projetos.

    Descrição: os projetos Java ME criados no NetBeans não podem ser executados em locais de Linux em japonês ou chinês com o JDK 6 instalado.

    Solução alternativa: use o JDK 5.0 ou execute os projetos de locais em inglês.

  • Problema nº 139031 do NetBeans: o emulador não se inicia no Windows se o binário Java não estiver no caminho do ambiente.

    Descrição: o Sun Java Wireless Toolkit e outros emuladores não se iniciam no Windows se Java.exe não estiver no caminho.

    Solução alternativa: instale o JDK usando as configurações padrão ou, se estiver realizando uma instalação personalizada, certifique-se de selecionar o JRE público para a instalação. Se o JDK já estiver instalado, você pode adicionar o arquivo Java.exe às configurações de caminho do sistema.

Problemas conhecidos: SOA


Esta versão de NetBeans SOA 6.5 permite explorar e avaliar a nova funcionalidade no IDE. Pedimos que você participe e nos dê sua opinião, registrando quaisquer problemas encontrados no sistema de rastreamento de problemas em https://netbeans.org/community/issues.html.

Os problemas de SOA não-solucionados nesta versão são os seguintes:

  • Para usar o Sun Java System Access Manager, você precisa instalar o download completo do IDE e a Java Application Platform SDK Atualização 3 Beta ou posterior, a partir do endereço http://java.sun.com/javaee/downloads/index.jsp.
  • Problema nº 114171: arquivos WSDL editados manualmente pelo CASA lançam a exceção de ponteiro nulo na saída build.xml.

    Descrição: o CASA não oferece suporte à edição do usuário de seus arquivos gerados e não fornece nenhum aviso quando um usuário realiza esse procedimento.

    Solução alternativa: não edite arquivos WSDL gerados pelo CASA. A edição manual de arquivos WSDL não é compatível e causará resultados inesperados.

  • Problema nº 84622: o IDE não reverte a recuperação do(s) arquivo(s) WSDL após cancelar a caixa de diálogo PartnerLink.

    Descrição: se você cancelar a caixa de diálogo PartnerLink que é exibida ao arrastar um arquivo WSDL ou um nó do serviço Web de um projeto NetBeans para um diagrama, o IDE não reverte a recuperação do(s) arquivo(s) WSDL.

    Solução alternativa: se esses arquivos não forem necessários para o projeto, basta excluí-los manualmente da janela Projetos, como você faria com qualquer outro recurso de projeto.

  • Problema nº 112344: a ação de arrastar e soltar no modo de exibição em árvore do editor de esquemas XML pode lançar exceções.

    Solução alternativa: use JDK 1.6.0_02 ou posterior.

  • Problema nº 96237: o XSLT Designer não oferece suporte a mensagens definidas em tipos complexos, simples ou embutidos.

    Descrição: nesta versão, o XSLT Designer não oferece suporte a mensagens declaradas em WSDL usando o atributo type="...". Somente as declarações com o atributo element="..." são compatíveis.

Problemas conhecidos: C/C++


O NetBeans 6.5 C/C++ oferece suporte a aplicativos C e C++ nos sistemas operacionais Microsoft Windows, Macintosh OS X, Linux e SolarisTM. Os desenvolvedores de C/C++ podem trazer os aplicativos existentes para o sistema de projeto NetBeans ou desenvolver novos aplicativos e bibliotecas. O suporte para C/C++ do NetBeans abrange o ciclo de desenvolvimento completo: projetos e modelos, um editor com reconhecimento de linguagem avançado, construções baseadas em makefile e depuração.

Para obter informações sobre a instalação e a configuração dos compiladores e das ferramentas para suporte do NetBeans C/C++, consulte Instalando e configurando o suporte para C/C++.

Os problemas C/C++ não-solucionados nesta versão são os seguintes:

  • Problema nº 108204: nas plataformas Windows, a entrada do usuário não funciona na janela Saída durante a depuração

    Descrição: nas plataformas Windows, quando estiver depurando um aplicativo e o tipo de console do projeto estiver definido como a janela Saída, não é possível digitar a entrada na janela Saída. Esse problema não afeta o desenvolvimento que usa um host remoto.

    Solução alternativa: defina o tipo de console como Terminal externo clicando com o botão direito do mouse no nó do projeto na janela Projetos, selecionando o nó Executar na caixa de diálogo Propriedades do projeto, selecionando Terminal externo na lista suspensa Tipo de console e clicando em OK.

  • Problema nº 121776: ocasionalmente, o código Sun Studio compilado tem problemas para ser depurado com gdb

    Descrição: nos sistemas operacionais Solaris e Linux, o código compilado com os compiladores Sun Studio nem sempre pode ser depurado corretamente com o depurador gdb. A maioria dos problemas conhecidos está corrigida no software Sun Studio 12, mas existem muitos outros. Se você tiver problemas para depurar programas compilados com compiladores Sun Studio 12, a primeira coisa a fazer é aplicar os patches do produto Sun Studio 12 mais recentes. Se o problema persistir, registre o defeito em Issuezilla e torne essa tarefa dependente da solução.

    À medida que os problemas forem encontrados, abriremos novos problemas e atualizaremos esta tarefa de acordo com eles, fornecendo algumas idéias do estado atual da depuração de programas compilados com compiladores Sun Studio 12 usando NetBeans e o depurador gdb.

    Esses problemas não afetam a depuração do código compilado com compiladores Sun Studio 12 usando o IDE Sun Studio 12 e o depurador dbx.

  • Problema nº 121775: é necessário o Xcode mais recente para depurar o Tiger (Mac OS X 10.4)

    Descrição: ao depurar no Macintosh com Tiger (Mac OS X 10.4), a versão padrão dos compiladores gcc e g++ gera o código com um bug na saída da depuração. A Apple lançou um pacote Xcode atualizado (versão 2.5) com uma correção para esse problema. As versões corretas de gcc, g++ e gdb são:

    gcc: 4.0.1 - Apple versão 5367
    g++: 4.0.1 - Apple versão 5367
    gdb: 6.3.50 - Apple versão 573 (10/20/2006)

    Para fazer o download dessa versão de Xcode, consulte http://developer.apple.com/tools/download.

    Esse problema não existe no Leopard (Mac OS X 10.5).

    Os sintomas do problema são que os executáveis compilados com o flag -gdwarf (o padrão para projetos de aplicativo em C/C++) têm informações de linhas incorretas e os pontos de interrupção talvez devam ser ignorados.
